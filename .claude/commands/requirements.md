# 要件定義ワークフロー

プロジェクトの要件定義を体系的に行うワークフローです。

## 使い方

````bash
/wf-requirements プロジェクトの概要や目的
```bash

---

## STEP 1: 作業ログ作成

`docs/tasks/ai-logs/YYYY-MM-DD_{slug}.md` に作業ログを作成すること。

**テンプレート:**

```markdown
# {タイトル}

## 概要

{作業内容の説明}

## 作業内容

{実施した内容を記録}

## 検証結果

{テストや動作確認の結果}

---

## 発生した問題・はまったこと

### 問題1: {問題の概要}

**現象**:
**原因**:
**解決策**:
**今後の注意点**:

## 振り返り

### 学んだこと

-

### ドキュメント・ルール更新の提案

| 対象ファイル | 更新内容 |
| ------------ | -------- |
|              |          |
```bash

## STEP 2: 背景・目的の整理

以下を明確にすること:

- プロジェクトの背景（なぜ必要か）
- 解決したい課題
- 達成したいゴール

## STEP 3: ユーザー・ステークホルダーの特定

以下を定義すること:

- 主要ユーザー（ペルソナ）
- ステークホルダー
- 各者の期待・要求

## STEP 4: 機能要件の定義

以下を整理すること:

- 必須機能（Must Have）
- あると良い機能（Should Have）
- 将来対応（Could Have）

各機能について:

- ユーザーストーリー形式で記述
- 受け入れ条件（Acceptance Criteria）を定義

## STEP 5: 非機能要件の定義

以下を検討すること:

- パフォーマンス要件
- セキュリティ要件
- 可用性・信頼性
- 保守性・拡張性
- 対応プラットフォーム・環境

## STEP 6: 制約条件の整理

以下を明確にすること:

- 技術的制約
- ビジネス的制約（予算、スケジュール）
- 法的・規制上の制約

## STEP 7: 優先順位付け

機能要件に対して優先順位を決定すること:

- MoSCoW 法または他の手法で優先度を設定
- MVP（Minimum Viable Product）の範囲を定義

## STEP 8: ユーザー確認

**要件定義書をユーザーに提示し、承認を得ること。**

## STEP 9: 設計タスクの作成

要件定義から設計フェーズのタスクを洗い出し、`docs/tasks/pre-release/index.md` に追加すること。

### 追加するタスク例

- 技術スタック・アーキテクチャ設計（`/design`）
- 開発環境のセットアップ（`/setup`）
- 各機能の詳細設計（`/design-doc`）

### 記載フォーマット

```markdown
## 📋 TODO

### 設計フェーズ

- [ ] 技術スタック決定
- [ ] 環境セットアップ
- [ ] ユーザー API 設計
- [ ] DB スキーマ設計
- [ ] ログイン画面設計
```

### ポイント

- 必須機能（Must Have）を優先
- シンプルなチェックリストで十分（ID 連番は任意）

## STEP 10: 完了

1. 作業ログの「振り返り」セクションを記入すること
2. 要件定義書を `docs/specs/` に保存すること
3. タスクリストが更新されていることを確認すること
4. Git コミットすること
5. ユーザーに完了報告すること
````
