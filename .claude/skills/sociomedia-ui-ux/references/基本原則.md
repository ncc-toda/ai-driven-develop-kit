# 基本原則（原則 1-3）

## 1. シンプルにする (Simple) [P0]

**定義**: 機能や情報を厳選し、できるだけ要素を少なくする

- **80/20 ルール**: ユーザーの 80% は機能の 20% しか使わない
- **情報階層**: 一次（必須）→ 二次（参照）→ 三次（詳細）
- **プログレッシブディスクロージャー**: 最初は最小限、操作に応じて詳細表示
- **スマートデフォルト**: 最も一般的な選択肢を初期値に

**良い例**: Google 検索（検索ボックスと検索ボタンのみ）、段階的開示フォーム

**悪い例**: 10 個以上のウィジェットが同時表示されるダッシュボード、全機能に同じ視覚的重要度

**チェック項目**:

- [ ] このページに不要な要素はないか？
- [ ] 削除できる機能はないか？
- [ ] 情報の優先順位は明確か？
- [ ] ユーザーが一度に処理する情報量は 7±2 項目以内か？

---

## 2. 簡単にする (Easy) [P0]

**定義**: 利用方法を効率化し、目的達成までの手順と労力を減らす

- **3 クリックルール**: 重要な情報・機能は 3 クリック以内でアクセス
- **ワンクリックアクション**: お気に入り登録、いいね等は確認不要
- **自動入力**: 郵便番号 → 住所、カード番号 → カード会社自動判別
- **入力項目削減**: 必須項目のみに絞る

**良い例**: Amazon 1-Click 購入、段階的登録（メール+パスワードで開始、追加情報は後で）

**悪い例**:

```text
❌ 複雑な会員登録:
  ステップ 1: 基本情報入力（10 項目）
  ステップ 2: メール認証
  ステップ 3: 詳細プロフィール（15 項目）
  ステップ 4: 利用規約同意（3 ページ）
  → 離脱率: 約 70%
```

**チェック項目**:

- [ ] タスク完了までのステップ数は最小か？
- [ ] 自動化できる部分はないか？
- [ ] ユーザーの過去の行動を活用できているか？
- [ ] ショートカットは用意されているか？

---

## 3. 一貫性 (Consistency) [P0]

**定義**: 配色、形状、配置、振る舞いなどに一貫したルールを適用する

### 一貫性の種類

1. **内部一貫性**: 同一アプリ内での統一
2. **外部一貫性**: 業界標準やプラットフォーム規約との統一
3. **視覚的一貫性**: タイポグラフィ、カラーパレット、アイコンスタイル
4. **機能的一貫性**: 操作や振る舞いの統一

### 実装ポイント

- **デザインシステム**: ボタン、色、サイズをトークンで定義
- **コンポーネントライブラリ**: 再利用可能な UI 部品
- **色の意味統一**: 青=情報、緑=成功、赤=エラー、オレンジ=注意
- **タイポグラフィ階層**: H1→H2→H3→Body→Caption

**良い例**: Apple iOS（すべてのアプリで共通のナビゲーションパターン）、Material Design

**悪い例**:

```text
❌ 一貫性のないサイト:
  ページ A: 青いボタンが「送信」
  ページ B: 青いボタンが「キャンセル」
  ページ C: 緑のボタンが「送信」
```

### 一貫性を破るべき時

- 重要な操作の強調（決済ボタンを目立たせる）
- 危険な操作の警告（削除ボタンを赤色に）
- 特別なコンテキスト（プロモーション期間中の特別デザイン）

**チェック項目**:

- [ ] 同じ機能のボタンは同じスタイルか？
- [ ] 色の使い方にルールがあるか？
- [ ] ナビゲーションパターンは統一されているか？
- [ ] エラーメッセージのトーンは一貫しているか？
