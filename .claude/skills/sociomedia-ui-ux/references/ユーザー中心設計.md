# ユーザー中心設計（原則 9-13）

## 9. ユーザーの主導権 (User Control) [P0]

**定義**: ユーザーがシステムをコントロールできるようにする

### 主導権を奪う典型例と改善

| 問題                           | 改善策                                           |
| ------------------------------ | ------------------------------------------------ |
| 強制チュートリアル             | 「後で見る」選択肢、いつでもヘルプから再確認可能 |
| スキップできないアニメーション | タップで完了、設定で無効化オプション             |
| 自動更新による作業中断         | 作業中は延期、明示的許可、自動保存で保護         |
| キャンセルできないプロセス     | いつでもキャンセル可能、一時停止・再開機能       |

### 実装ポイント

- **Exit**: ESC キーでモーダル終了
- **Undo/Redo**: コマンドパターンで操作履歴管理
- **自動保存**: インクリメンタル保存+手動保存も可能
- **通知コントロール**: 詳細な通知設定（頻度、種類）
- **カスタマイズ**: テーマ、レイアウト密度

## 10. モードレス (Modelessness) [P1]

**定義**: できるだけモードをなくし、ユーザーが自由な順序で作業を行えるようにする

### 悪いモードの例

1. **モーダルダイアログ**: 他の作業不可
2. **編集モード vs 閲覧モード**: モード切替が必要
3. **挿入モード vs 上書きモード**: 現在モードが不明確

### モードレス設計の方法

1. **パネル/ドロワー使用**: モーダルの代替
2. **インラインエディティング**: 直接編集可能
3. **並行作業**: タブベースで複数作業
4. **コンテキスト依存 UI**: 選択状態に応じたツールバー

### 良い例

- macOS システム設定: サイドバー選択、即座に適用、モーダルなし
- Gmail: メール作成画面が浮遊、受信メール確認しながら返信

### モードが許容される場合

- 重要な決断（削除確認）
- エラー/警告通知
- システム根本変更（更新）

## 11. ユーザーの言葉を使う (Use User's Language) [P1]

**定義**: システム内部の技術用語ではなく、ユーザーが普段使っている言葉を使う

### 言葉の見つけ方

1. ユーザーインタビュー
2. カスタマーサポートログ分析
3. ソーシャルメディアの言及
4. 検索クエリ分析
5. A/B テスト

### 翻訳例

| 技術用語              | ユーザー向け                 |
| --------------------- | ---------------------------- |
| Authentication failed | ログインできませんでした     |
| Invalid input         | 入力内容をご確認ください     |
| Database error        | 一時的に利用できません       |
| Cache cleared         | 最新の情報に更新しました     |
| Session expired       | もう一度ログインしてください |

**良い例**:

```html
<!-- ✅ 分かりやすいエラー -->
<p>写真は 10 枚まで添付できます</p>
<p>商品名は 40 文字以内で入力してください</p>
```

**悪い例**:

```html
<!-- ❌ 技術的すぎる -->
<p>HTTP 500 Internal Server Error</p>
<p>Invalid JSON format in request body</p>
```

## 12. ユーザーの記憶に頼らない (Don't Rely on Memory) [P1]

**定義**: 必要な情報は、それが必要となるその場で参照できるようにする

### Recognition > Recall

| Recall（想起） | Recognition（認識）      |
| -------------- | ------------------------ |
| 県名を入力     | 県名をプルダウンから選択 |
| コマンド入力   | ボタンクリック           |

### 記憶に頼らない実装

1. **文脈情報表示**: パンくずリスト、関連情報並列表示
2. **最近使った項目**: アクセス履歴、頻度・新しさでソート
3. **プレビュー機能**: 変更前後の比較
4. **サマリー確認画面**: 購入確認で全情報集約
5. **ツールチップ/ヘルプ**: コンテキスト依存

**良い例**: Amazon 購入プロセス（各段階で必要情報がその場で見える）、パスワード要件（入力フィールド横にリアルタイム表示）

**悪い例**: 複数画面にまたがる情報参照、エラー箇所の説明が別画面

## 13. エラーを回避する (Avoid Errors) [P0]

**定義**: エラーが起きないように予防する設計を最優先とする

### エラー回避の方法

1. **確認**: 危険操作は確認ダイアログ
2. **制約**: 無効な操作を物理的に不可能に
3. **警告**: 潜在的問題を事前に通知
4. **バリデーション**: リアルタイムで入力検証
5. **取り消し**: Undo 機能、ゴミ箱

### エラー回避の実装例

```html
<!-- 削除確認 -->
<dialog>
  <h3>⚠️ この操作は取り消せません</h3>
  <p>本当に削除しますか？</p>
  <button class="danger">削除する</button>
  <button autofocus>キャンセル</button>
</dialog>
```

### エラーメッセージのガイドライン

- **具体的**: 何が問題か明確に
- **建設的**: 解決方法を提示
- **平易な言葉**: 技術用語を避ける
- **非難しない**: ユーザーを責めない

**良い例**:

```text
✅ メールアドレスの形式が正しくありません。
   例: example@email.com
```

**悪い例**:

```text
❌ Error: Invalid email format
```
