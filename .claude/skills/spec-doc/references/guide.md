# 開発ドキュメントガイド

## 目的

開発時に作成・更新すべきドキュメントを整理し、記載内容と運用ルールを統一する。

## 作成すべきドキュメント（必須）

| ドキュメント | 目的                                             | テンプレート                  |
| ------------ | ------------------------------------------------ | ----------------------------- |
| UI/UX 方針   | プロダクト全体の UX 方針とデザイン原則を固定     | `ui-ux-concepts-template.md`  |
| 画面設計書   | 画面単位の UI/UX と挙動を仕様化                  | `screen-design-template.md`   |
| 機能設計書   | 機能単位の振る舞い・データ設計・非機能要件を定義 | `function-design-template.md` |

## 必要に応じて作成するドキュメント

- **アーキテクチャ方針（ADR）**: フォルダ構成、責務分離、状態管理、依存方向
- **データ設計書**: エンティティ、永続化、識別子、更新規則
- **API 設計書**: エンドポイント、リクエスト/レスポンス、エラーコード
- **例外・エラー辞書**: 共通エラー種別と表示/復帰導線
- **イベント命名・計測設計**: イベント名規約、必須パラメータ
- **テスト方針**: ユニット/統合/E2E の範囲、モック方針
- **リリース運用**: バージョニング、申請、ロールバック手順
- **セキュリティ/プライバシー方針**: 収集データ、ログマスキング
- **非機能要件**: パフォーマンス目標、オフライン方針
- **仕様変更履歴**: 変更点と背景の追跡
- **決定ログ（ADR）**: 重要な設計判断と代替案

## ドキュメント間の整合ルール

1. UI/UX 方針の原則・状態・フィードバック方針 → 画面設計書・機能設計書に必ず反映
2. 画面設計書の `interactions` / `interactionId` ↔ 機能設計書の画面対応
3. 画面設計書の `dataContract` ↔ 機能設計書の `dataContract`（同一名称/型）
4. 文言・用語は UI/UX 方針の「用語ルール」に揃える

## 記載時の注意

- 仕様は「何が起きるか/いつ起きるか」を明確に書く（実装手順や見た目説明は最小限）
- 状態名（`loading/empty/error/success`）は統一、再利用可能な用語は定義セクションに集約
- 外部依存（API/外部サービス/権限）や前提条件は冒頭に必ず記載
- アクセシビリティ要件（読み上げ/タップ領域/色の扱い）を省略しない

## 更新時の注意

- 画面追加・遷移変更・文言変更・データ契約変更があれば該当ドキュメントを必ず更新
- UI/UX 方針の変更が発生した場合、関連する画面設計書/機能設計書への影響を点検
- 変更点は Change Log に記録し、関連する決定ログがある場合はリンク
