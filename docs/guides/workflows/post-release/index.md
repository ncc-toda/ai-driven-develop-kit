# post-release ワークフロー

運用フェーズで使用するワークフローです。
リリース後の保守・改善作業をカバーします。

## ワークフロー一覧

| ファイル                           | 用途             |
| ---------------------------------- | ---------------- |
| [bug-fix.md](./bug-fix.md)         | バグ修正         |
| [feature-add.md](./feature-add.md) | 機能追加         |
| [refactor.md](./refactor.md)       | リファクタリング |

## ワークフロー選択ガイド

作業内容に応じて適切なワークフローを選択してください。

```text
┌─────────────────────────────────────────────────────────┐
│                    作業内容は？                          │
└─────────────────────────────────────────────────────────┘
                          │
          ┌───────────────┼───────────────┐
          ▼               ▼               ▼
   ┌────────────┐  ┌────────────┐  ┌────────────┐
   │ 不具合修正  │  │ 新機能追加  │  │ コード改善  │
   │ 動作不良    │  │ 機能拡張    │  │ 構造改善    │
   └──────┬─────┘  └──────┬─────┘  └──────┬─────┘
          ▼               ▼               ▼
   ┌────────────┐  ┌────────────┐  ┌────────────┐
   │  bug-fix   │  │ feature-add│  │  refactor  │
   └────────────┘  └────────────┘  └────────────┘
```

## 各ワークフローの概要

### bug-fix.md（バグ修正）

**使用場面**:

- 本番環境で不具合が発生した
- テストで問題が見つかった
- 期待どおりに動作しない箇所がある

**特徴**:

- 再現確認から開始
- バグを検出するテストを先に書く（TDD）
- 最小限の修正で対応

---

### feature-add.md（機能追加）

**使用場面**:

- 新しい機能を追加したい
- 既存機能を拡張したい
- ユーザーからの要望に対応したい

**特徴**:

- 既存スキーマ・構造との整合性を確認
- インターフェース設計から開始（SDD）
- TDD で実装

**注意**: 大規模な機能追加の場合は、pre-release の [design-doc.md](../pre-release/design-doc.md) + [impl-from-design.md](../pre-release/impl-from-design.md) の使用を検討してください。

---

### refactor.md（リファクタリング）

**使用場面**:

- コードの可読性を向上させたい
- 重複コードを整理したい
- 技術的負債を解消したい

**特徴**:

- 振る舞いを変えない
- 既存テストが十分にあることを確認
- 段階的に実施

## 環境整備について

環境整備（開発環境、CI/CD、依存関係の更新など）は [common/env-setup.md](../common/env-setup.md) を使用してください。
